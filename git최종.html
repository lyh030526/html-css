1주차 — Git & GitHub 기초 개념

버전 관리 시스템의 이해 · Git과 GitHub의 차이 · 핵심 동작 4단계
01
주간 요약 Git은 내 컴퓨터에서 코드의 변경 이력을 기록하는 버전 관리 시스템이고, GitHub는 이를 온라인에 올려 협업할 수 있게 해주는 플랫폼입니다. 이 도구들의 차이점과 코드를 수정하고 업로드하는 전체적인 흐름을 이해하는 것이 1주차의 목표입니다.

Git, GitHub, Branch 란?

코드 관리부터 협업까지 필수적인 핵심 개념 3가지입니다.

개념설명
📦 Git내 컴퓨터에서 코드의 변경 이력을 기록하고 관리하는 버전 관리 시스템입니다.
☁️ GitHubGit으로 관리하는 프로젝트를 올리는 온라인 저장소이자 협업 플랫폼입니다.
🌿 Branch기존 코드(main)와 분리되어 독립적으로 작업할 수 있는 공간입니다.

Git의 핵심 동작 흐름 4단계

파일을 처음 수정하고 GitHub라는 원격 저장소에 올리기까지의 가장 기본적인 4단계 과정입니다.

전체 흐름 구조도
📄 Modify
(파일 수정) → 🚩 Add
(Stage 영역) → 💾 Commit
(로컬 저장) → 🚀 Push
(GitHub 업로드)

2주차 — Git 기본 설정 및 필수 명령어

개념 최소화 · 환경 설정 · add / commit / log 흐름
02
주간 요약 Git은 단순한 "저장"이 아니라 파일 변경 이력을 관리하며 시간별 스냅샷을 남기는 도구입니다. 기본 사용자 정보를 설정하고, 필수 명령어 4개를 활용한 작업 흐름을 반복하는 것이 핵심입니다.

기본 설정 및 명령 구조

Git은 누가 작업했는지 기록하는 도구이므로 사용자 이름과 이메일 설정이 필수입니다. 명령어 기반 도구이며, 모를 때는 git help를 활용할 수 있습니다.

코드 & 결과
# 사용자 정보 설정 git config --global user.name "이름" git config --global user.email "이메일" # 설정 확인 cat ~/.gitconfig # 도움말 확인 git help git help <명령어>
개념 이해하기
[user]
    name = 홍길동
    email = user@example.com

설정된 정보는 이후 생성되는 모든 커밋(Commit) 기록에 작성자로 남게 됩니다.

필수 명령어 4개와 작업 흐름

commit은 게임의 세이브 포인트처럼 "현재 상태 저장 + 설명"을 의미합니다. 파일 수정 → add → commit 흐름을 반복하는 것이 Git 사용의 핵심입니다.

코드 & 흐름 이해
# 1. 저장소 시작 git init # 2. 파일 추가 (커밋 대기 상태로) git add . # 3. 상태 저장 및 설명 추가 git commit -m "메시지" # 4. 기록 확인 git log
작업 흐름 반복
파일 수정 → git add → git commit
이 흐름을 반복!

3주차 — Git 상태 공간 및 협업 시작

WD / Stage / Repo · git log 활용 · clone / pull
03
주간 요약 파일이 존재하는 3가지 상태 공간(Working Directory, Staging Area, Repository)을 이해하는 것이 가장 중요합니다. 또한, Git은 기록을 보는 도구로서의 역할이 크며, clone과 pull을 통해 협업의 기본을 배웁니다.

Git 3가지 상태 공간 구조

하나의 파일이 수정된 버전과 스테이지된 버전 등 동시에 2개 상태로 존재할 수 있습니다. "Git은 저장이 아니라 상태 변화와 기록을 관리하는 시스템이다"라는 점을 기억해야 합니다.

상태 (공간)의미
Working Directory내가 현재 파일을 수정하고 작업하는 곳
Staging Area커밋(저장) 대기 공간
Repository완전히 저장된 기록 (Commit이 완료된 곳)

git log (기록 확인)

로그 기록에는 Commit ID, 작성자, 날짜, 메시지가 포함됩니다. 목적에 맞게 옵션을 추가해 기록을 효율적으로 확인할 수 있습니다.

명령어 비교
# 전체 기록 상세 보기 git log # 한 줄로 요약해서 보기 git log --oneline # 그래프 형태로 시각화하여 보기 git log --graph
결과 미리보기 (개념적)
* a1b2c3d (HEAD -> main) 세 번째 커밋
* 8f9e0a1 두 번째 커밋 메시지
* 5d4c3b2 첫 저장소 초기화

clone과 pull (협업의 시작)

처음 프로젝트를 가져올 때는 clone, 이후 업데이트된 내용을 가져올 때는 pull을 사용합니다.

코드 & 결과
# GitHub에서 내 컴퓨터로 가져오기 git clone URL # 이후 변경된 업데이트 내용 가져오기 git pull
핵심 차이
명령어의미
clone처음 가져오기
pull업데이트 가져오기

4주차 — 원격 저장소 (GitHub)

로컬과 원격 · push 흐름 · 많이 하는 실수
04
주간 요약 내 컴퓨터(로컬 저장소)와 GitHub(원격 저장소)의 개념을 연결합니다. Git은 혼자 쓰는 것이 아니라 "공유"하려고 쓰는 도구임을 명심하며 push/pull 흐름을 익힙니다.

원격 저장소 연결 (clone과 remote)

clone을 하면 새로운 폴더와 .git 저장소가 생성되며 git init이 필요 없습니다. 또한 origin이라는 기본 원격 저장소 이름이 자동으로 생성되어 연결까지 끝난 상태가 됩니다.

코드 & 결과
# 처음 복제하기 git clone URL # 원격 저장소 연결 상태 확인 git remote git remote -v
결과 미리보기 (remote -v)
origin https://github.com/user/repo.git (fetch)
origin https://github.com/user/repo.git (push)

인증을 위해 GitHub는 비밀번호 대신 PAT(토큰)를 사용합니다.

가장 중요한 push 흐름

내 작업 결과를 GitHub에 업로드하는 명령어입니다. Commit이 없으면 push를 할 수 없습니다. 또한, 충돌 방지를 위해 항상 작업 전에는 pull을 먼저 해야 합니다.

코드 & 흐름
# 1. 파일 수정 후 추가 및 커밋 git add . git commit -m "수정" # 2. GitHub에 업로드 git push
전체 실습 흐름 요약
clone → 작업 → commit → push
초보자가 많이 하는 실수
  • pull 안 하고 작업: 충돌 발생 가능
  • commit 안 하고 push: 업로드 안 됨
  • clone 안 하고 init: 원격 연결 안 됨

5주차 — 브랜치(Branch)와 협업(PR)

작업 공간 분리 · PR 요청 · Merge 흐름
05
주간 요약 안전한 협업을 위해 main 브랜치는 직접 건드리지 않고, 새로운 브랜치를 만들어 독립된 공간에서 작업합니다. 작업 완료 후 PR(Pull Request)을 통해 합쳐달라고 검토를 요청하는 것이 협업의 핵심입니다.

브랜치(Branch) 생성 및 이동

브랜치별로 작업을 분리하면 충돌을 감소시킬 수 있습니다. 브랜치를 만들고 반드시 그 안으로 이동(checkout)하여 작업해야 합니다.

코드 & 결과
# 브랜치 생성 후 이동 git branch feature1 git checkout feature1 # 위 두 명령어를 한 번에 실행 git checkout -b feature1 # 작업 후 브랜치 이름으로 push git push origin feature1
협업 개념 요약
개념의미
branch작업 공간 분리
push작업 업로드
PR합쳐달라고 요청 (Merge 요청)
merge실제 반영

PR (Pull Request) 과 전체 흐름

PR은 GitHub에서 "이 작업을 main에 합쳐주세요"라고 검토를 요청하는 과정입니다. 기존의 복잡한 Fork 방식보다 Branch + PR 방식이 초보자에게 더 쉽고 추천됩니다.

전체 한 줄 흐름 (학생용 암기)
브랜치 만들기 → 작업 (add/commit) → push → PR (요청) → merge (반영)

이 전체 흐름을 이해하는 것이 5주차 협업 과정의 최종 목표입니다.
